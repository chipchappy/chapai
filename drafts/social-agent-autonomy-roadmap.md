# Social Agent Autonomy Roadmap

## Goal
Turn `Mercury` into a real growth operator that can:
- package content
- study creator trends
- maintain outreach lists
- prepare post candidates
- publish automatically only when platform connectors are truly ready

## Operating modes

### 1. Manual-ready
Use this now when no platform API credentials are available.

Mercury can still:
- maintain a reply bank
- maintain creator shortlists
- package X and Instagram posts
- study search intent
- prepare warm outreach sequences

### 2. Semi-auto
Use this when one or more platform connectors exist but publishing is not fully trusted.

Mercury can:
- prepare post candidates
- rank them by urgency / relevance
- keep a trend watchlist
- hand exact ready-to-post copy to the operator
- log what themes are performing best

### 3. API-ready
Use this only when:
- X posting credentials are live
- Meta / Instagram professional publishing credentials are live
- safe posting limits are defined
- post review rules are explicit

Then Mercury can:
- publish bounded approved posts
- schedule recurring founder-safe updates
- collect performance feedback
- learn which topics produce replies, clicks, and creator responses

## Connector requirements

### X
- official write-capable app credentials
- access token and secret
- rate / cadence rules
- one reviewed content queue

### Instagram
- professional account path
- Meta Graph / publishing setup
- access token management
- one approved publishing lane

## Learning loop
Every cycle Mercury should update:
- top-performing topics
- top-performing hooks
- creators who replied
- creators worth a second pass
- posts that generated clicks or demos

## Safe posting rules
- no spam
- no fake testimonials
- no posting while blocked on account auth
- no more than one direct CTA in a short reply
- prefer warm, useful, human-sounding language

## Next unlock
The next practical unlock is not “full autonomy.” It is:
1. maintain a better reply queue
2. maintain a creator watchlist
3. keep post candidates fresh
4. add platform credentials only when the workflow is stable
